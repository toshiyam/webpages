# iseten

異世界転生・人生観測放置ゲーム。異世界へ送り込んだ転生者が、性格・能力・環境・人間関係に基づいて自律的に人生を歩む様子を観測するブラウザゲームです（issue #1, #3, #5, #7, #9, #10, #11, #14, #15, #17, #18, #19, #20, #21, #22, #23, #24, #25）。

プレイヤーは転生者を直接操作せず、転生条件を与えた人物が期待どおりにも期待外れにも生きていく過程を見届けます。本格的なAI・LLM・機械学習は使用せず、性格・状況・少量の乱数を用いた重み付き意思決定でゲームを進行します。

## ディレクトリ構成

将来のiOSネイティブアプリ化を見据え、ゲームデータ・ゲームエンジン・UIを分離しています。

```text
game-data/       ゲームデータ（JSON）
  traits.json      基礎能力6種・性格軸10種の定義
  occupations.json 職業10種の定義・収入・危険度
  world.json       地域名・NPC名プール・死因ラベル・世界状態の初期値としきい値
  elements.json    転生者固有の特殊要素9種（+「特別な要素なし」）
  goals.json       生涯目標12種の定義
  items.json       転生準備で持ち込める持込アイテム8種の定義
  skills.json      転生準備で選べる初期スキル8種の定義
  burdens.json     転生準備で任意に選べる制約・代償6種の定義
  events.json      イベント66件（鍛冶屋/冒険者/商人などの3系統＋人生アーク10系統のイベント連鎖を含む）

game-engine/     ゲームロジック（状態を入力し、新しい状態とログを返す関数群）
  rng.js                   乱数・重み付き抽選などの汎用ユーティリティ
  character-generator.js   転生者のランダム生成（特殊要素の付与を含む）
  discovery.js             転生記録図鑑の発見カウンタ・段階解禁の条件評価・死亡時スナップショット生成、既存セーブ移行時の発見状態復元（issue #9, #15）
  starting-grants.js       転生準備（持込アイテム・初期スキル・制約）の適用と消費管理、discoveries渡し時の段階解禁の強制（issue #7, #9）
  event-selector.js        イベントの発生条件判定・重み付き抽選（世界状態/生涯目標由来のフラグ生成、weightContextBonusによる発生頻度補正を含む）
  decision-engine.js       性格・能力・状況に基づく選択肢の重み付き抽選、requiredFlags/excludedFlagsによる選択肢の解禁・封印（全選択肢封印時に無視するフォールバックは行わない）
  effect-processor.js      選択結果の能力・職業・所持金・人間関係・生涯目標・世界状態・持込アイテム消費・人生の特殊な結末（endLife）への反映
  mortality.js             死亡判定・死因決定
  life-rank.js             人生アークの結末タグから人生ランク（伝説/災厄/平穏/著名/平凡）を判定
  consistency.js           自己テスト・静的チェック・人生ごとの整合性チェック（issue #5, #7, #9, #10, #11, #14, #15）
  time-processor.js        1年分の時間経過（世界状態の自然変動を含む）をまとめて処理する simulateYear()、不老不死等の特殊な結末（endLife）の処理
  summary-generator.js     死亡時の人生要約テンプレート生成（目標・特殊要素・転生準備の持込・使用結果・世界への影響・特殊な結末を含む）
  simulate.js              UIを介さず人生を高速実行するバランス検証用関数、同一候補・付与内容比較試験、転生記録図鑑の発見・解禁の巻き戻り検証
  load-data.js             game-data/*.json を1つのデータバンドルへ正規化
  index.js                 上記モジュールの再エクスポート

web-ui/          ブラウザ表示・入力・画面遷移
  main.js          状態管理・localStorage保存（スキーマ移行を含む）・画面描画・イベントハンドラ
  styles.css       スタイル

scripts/
  simulate-cli.js         Node上でUIを介さずバランス検証を実行するCLI
  build-single-file.mjs   game-data/game-engine/web-ui/index.html から配布用の単一HTMLを自動生成する

index.html       開発確認用のブラウザエントリーポイント
```

game-engine 配下の関数はいずれも DOM に依存しない純粋関数に近い形にしてあり、ブラウザ版（`web-ui/`）でも Node 版（`scripts/simulate-cli.js`）でも同じロジックを利用します。

## 動かし方

### ブラウザで動かす（開発確認用）

`index.html` は `game-data/*.json` を `fetch` で読み込むため、`file://` では動作しません。リポジトリ直下で簡易サーバーを起動してください。

```bash
npx serve .
# または
python3 -m http.server 8080
```

起動後、表示されたURLをスマートフォン含むブラウザで開いてください。

### 配布用の単一HTMLをビルドする

`toshiyam/webpages` で配布しているスマホ対応の1ファイル版（`tensei-life-watch.html`）は、このリポジトリのソースから自動生成しています。手作業で二重に書き直さないよう、更新のたびに以下を実行してください。

```bash
node scripts/build-single-file.mjs            # ./tensei-life-watch.html を生成
node scripts/build-single-file.mjs out.html    # 出力先を指定する場合
```

生成したファイルをそのまま `toshiyam/webpages` リポジトリの `tensei-life-watch.html` へコピーします。

### バランス検証（UIを介さない高速シミュレーション）

```bash
node scripts/simulate-cli.js 500   # 500人生を高速実行
```

平均寿命・結婚率・最終職業分布・死因分布・未発生イベントに加え、issue #3 のバランス目安（生涯目標形成率・人生アーク突入率／到達率・世界影響率・「ほぼ何も起きない人生」率・人生ランク分布）を出力します。

## 公開前チェックリスト（issue #22）

`toshiyam/webpages` への反映を含むリリース作業は、以下の順序で毎回同じ手順を踏む。抜け漏れを防ぐため、リリースPRを作成する前にこのチェックリストを上から順に実行すること。

### 1. 単一HTMLをビルドする

```bash
node scripts/build-single-file.mjs
```

- `game-data/*.json`・`game-engine/*.js`・`web-ui/*`・`index.html` から `tensei-life-watch.html` を再生成する（詳細は上記「配布用の単一HTMLをビルドする」を参照）
- 生成物のサイズやdiffを確認し、想定外の巨大化・内容の欠落がないか目視で確認する

### 2. 最低限実行する検証コマンド

```bash
node scripts/simulate-cli.js 1000
```

- 自己テスト（39項目）がすべてPASSすること
- 「合計整合性違反件数」が0件であること（暦の巻き戻り・死亡ログ後の継続・進捗と状態の矛盾など、既存の全検証項目を含む）
- 通常の開発中の動作確認は300〜500件で十分だが、公開前の最終確認では1000件で実行する

### 3. セーブデータ移行の確認

- 現在の`SCHEMA_VERSION`より古い`schemaVersion`のセーブ（`localStorage`に手動で仕込むか、過去バージョンのビルドで作成したもの）を読み込ませ、`loadState()`のマイグレーションでエラーにならず、警告バナー（「セーブデータを最新の形式に更新しました。」）と共に起動できることを確認する
- `discoveries`（発見・解禁状況）や`pastLives`（過去人生記録）のような蓄積系データが、移行や再読み込みで巻き戻ったり消えたりしないことを確認する（issue #15の`restoreDiscoveriesFromPastLives`が関係する領域）

### 4. スマホ幅の確認

- 生成した単一HTMLを実ブラウザ（Playwright推奨）で320px幅・375px幅の両方で開き、すべてのタブ（観測・人生記録・世界・図鑑）で横スクロールが発生しないことを確認する
- 主要な操作導線（転生開始→観測→死亡→次の転生、初期化モーダル、図鑑の絞り込みなど）を一通り操作し、レイアウト崩れがないことを確認する

### 5. 外部依存なしの確認

- ビルドした`tensei-life-watch.html`を`file://`で直接開いても動作すること（`fetch`によるCDN・APIへの通信が一切無いことの確認。開発用の`index.html`は`game-data/*.json`を`fetch`するため簡易サーバーが必要だが、ビルド後の単一HTMLは全データ・全スクリプトを埋め込み済みで`fetch`呼び出しを持たない）
- ブラウザの開発者ツールでネットワークタブを開いた状態で操作し、外部へのリクエストが発生しないことを確認する
- LLMや外部APIを一切使用していないこと（人生要約はテンプレート文の組み合わせのみで生成している）を確認する

### 6. `toshiyam/webpages` への反映手順

`toshiyam/iseten`と`toshiyam/webpages`、2つの別リポジトリのクローン（作業ツリー）を使う。以下の各ステップは、どちらのリポジトリで実行するかを明記している。両リポジトリとも、開発用ブランチ（例: `claude/iseten-issue-webpages-zt7ngq`）は使い捨てず、リリースのたびに`origin/main`へ強制同期して使い回す（新しいブランチをissueごとに作らない）。

**`git reset --hard`はその作業ツリーの未コミット変更・ステージ内容を問答無用で破棄する。** 手順の途中でこれを実行すると、直前にコピーした反映内容や検証結果を丸ごと失う。そのため、各リポジトリでの`reset --hard`は、そのリポジトリへ何かをコピー・変更する**前**に済ませておく（後からリセットしない）。実行前に`git status`でそのリポジトリに巻き込みたくない変更が残っていないかを必ず確認すること。

**(iseten)**

1. 対象issueの`toshiyam/iseten`側PRがマージされたことを確認する
2. `git branch --show-current`で現在のブランチを確認し、開発用ブランチ（例: `claude/iseten-issue-webpages-zt7ngq`）でなければ`git checkout <開発用ブランチ>`で切り替える。**この後の`reset --hard`は必ず開発用ブランチ上で行い、`main`ブランチ上では絶対に行わない**
3. `git status`で作業ツリーがクリーンであることを確認する
4. `git fetch origin main && git reset --hard origin/main`でマージ済みの最新`main`に同期する

**(webpages)**

5. `git branch --show-current`で現在のブランチを確認し、開発用ブランチでなければ`git checkout <開発用ブランチ>`で切り替える。**この後の操作は必ず開発用ブランチ上で行い、`main`ブランチ上では絶対に行わない**
6. `git fetch origin main`で最新`main`を取得する（この時点ではまだ`reset --hard`しない）
7. **push方法をここで判定し、後から新規変更を加えた後の差分と混同しない。** `git diff origin/<開発用ブランチ> origin/main`を確認する。空（＝リモートの開発用ブランチの内容が既に`main`へ取り込み済み）であれば、後述の手順14で`--force-with-lease`を使ってよい。空でない場合（まだ`main`に取り込まれていないリモート側の変更が残っている）は、その内容を確認し、失ってよいものでなければ上書きしない
8. `git status`で作業ツリーがクリーンであることを確認し、`git reset --hard origin/main`で最新`main`に同期する（コピー・ビルドより前に行うのが重要。後回しにすると次の手順以降の変更を消してしまう）
9. 手順1でマージ済みの`iseten`から、変更されたソースファイルを`webpages`側のソース同期先（`tensei-life-watch-src/`）へ`cp`で反映する
10. `node scripts/build-single-file.mjs`を`tensei-life-watch-src/`内で再実行し、単一HTMLを再ビルドする
11. 上記1〜5の検証（自己テスト・整合性チェック・セーブ移行確認・スマホ幅確認・外部依存なし確認）を、今回`webpages`向けに生成した成果物に対しても再実行する（同じビルドスクリプト・同じソースを使っていても、実際に生成された成果物で確認する）
12. ビルドしたHTMLを`webpages`リポジトリのルートへ配置し、`tensei-life-watch-src/`内に残ったビルド成果物は削除する（ソースディレクトリにビルド済み成果物を混在させないため）
13. `git status`/`git diff`で、想定したファイル（同期対象のソースとルートの単一HTML）だけが変更されていることを確認してから、変更をcommitする
14. `git push -u origin <開発用ブランチ>`する。手順7で「空」と判定していた場合は`--force-with-lease`を使ってよい（今回追加した新規コミットの有無は判定基準に含めない。判定は手順7の時点、つまり新規変更を加える前に済ませてある）。手順7で「空でない」と判定していた場合は、まず通常のpush（force無し）を試み、拒否されたら手順7で確認した内容を再度検討してから判断する
15. draft PRを作成する

### 7. リリースPR本文に書くべき内容

- **対応Issue**: 対応するissue番号とタイトル
- **変更概要**: 何をどう変えたか（箇条書き）
- **仕様上の判断**: 複数の実装方針がありえた場合に、なぜその判断をしたか
- **保存互換性への影響**: 既存セーブの構造・スキーマに影響があるか、無いか
- **実行した自動テスト**: `simulate-cli.js`の実行結果、Playwrightで確認した項目
- **実ブラウザ確認**: 確認したビューポート幅（320px/375pxなど）と確認内容
- **検証画像の保存先**: `docs/verification/pr-<PR番号>/`（`toshiyam/iseten`側PRのみ。screenshotは`toshiyam/webpages`側にはコピーしない）
- **未確認事項**: 実機（実デバイス）でのタップ操作など、今回検証できなかった項目があれば明記する

## 人生要約・統計値の意味的整合性との対応（issue #5）

issue #3 実装への敵対的検証（PR #4 のレビュー）で見つかった意味的な不正確さに対応しています。

- **無関係な目標が誤って「達成」扱いになる問題**: `effects.goalStatus` が現在の `character.goal.id` を確認せずに適用されていたため、例えば目標が「家族を守る」の転生者が勇者アークの結末に到達すると、無関係な目標が達成扱いになっていた。`effects.goalResolution: { ids, progress, status }` に置き換え、`character.goal.id` が `ids` に含まれる場合のみ適用するよう修正（`effect-processor.js` の `applyGoalResolution`）
- **達成なのに進捗100%未満という矛盾**: `status: "completed"` を適用しても進捗値をそろえていなかったため、「状態=達成、進捗=30%」のような矛盾した組み合わせが生成され得た。`applyGoalResolution` が `completed` を適用する際に `progress` を必ず100へ確定するよう修正
- **世界への影響の誤帰属**: 死亡時要約が `state.world`（転生をまたいで持続し、自然ドリフトも含む）と初期値の差分をそのまま「この人生の影響」として記述していたため、前の転生者の遺産や乱数による揺らぎまで誤って書かれていた。転生者ごとにイベント効果による増減だけを積算する `character.worldImpact` を新設し、要約とバッチ統計の両方をこれだけを参照するよう修正
- **決着経路のない生涯目標**: 「家族を守る」「世界の果てを見る」「復讐を遂げる」「不老不死になる」「元の世界へ帰る」「魔法を解明する」の6目標に専用の決着イベント（`goal_check_*`）を追加し、12種すべての目標に達成/失敗/変質へ至る経路ができた
- **性格重みに能力値キーが紛れて無効化していた問題**: `traitWeights` に `willpower` / `social` のような能力値のキーが指定されており、`character.traits` に存在しないため黙って無視されていた。`decision-engine.js` に `abilityWeights` を追加し、該当箇所を正しい方へ移設

### 検証機能（`game-engine/consistency.js`）

- **自己テスト**（`runGoalResolutionSelfTests`）: 「対象外の目標は完遂されない」「一致する目標のみ完遂される」「目標が無ければ無視される」「completedは進捗100を保証する」など、乱数シミュレーションでは再現しづらい境界条件を直接検証する。CLI・ブラウザ検証パネルの実行のたびに毎回走る
- **静的チェック**（`runStaticConsistencyChecks`）: `game-data/events.json` 全体に対して、`traitWeights` への能力IDの混入、`goalResolution.ids` の指定漏れ（指定を忘れると「現在の目標が何であれ状態変更される」という問題1の再発経路になる）を検出する
- **人生ごとの整合性チェック**: 「状態=達成なのに進捗100%未満」（`findGoalProgressViolation`）を1,000人生シミュレーションで確認する
- **目標別統計**: 12目標それぞれの形成数・完遂数・失敗数・放棄数・変質数・未決着数・平均進捗・「決着イベントに一度も到達できなかった人数」を集計。CLI (`scripts/simulate-cli.js`) とブラウザの「開発者向け: バランス検証」パネルの両方で確認できる
- **世界影響統計**: 自然ドリフトを含まない `character.worldImpact` のみを使った影響率・項目別平均寄与・正負の人数

## 転生準備フェーズとの対応（issue #7）

転生候補確定後、開始前に「持込アイテム」「初期スキル」「制約・代償」からそれぞれ最大1つを選ぶ転生準備フェーズを追加しました。プレイヤーが選ぶのは初期条件だけで、その後どう使う（あるいは使わない）かは転生者自身の性格・状況・乱数に委ねられます。

- **持込アイテム**（`game-data/items.json`、8種）: 単なる能力値加算ではなく、`item_<id>` フラグ経由で既存の `contextWeights` / `requiredFlags` / `excludedFlags` に接続。医療箱・封印された指輪・銀貨袋・名もなき種子の4種には専用または明確な活用イベント（`illness_common` の看病選択肢、`item_sealed_ring_awakening`、`item_silver_purse_targeted`、`item_nameless_seed_harvest`）があり、消費（`usesRemaining`）や封印・喪失も表現できる
- **初期スキル**（`game-data/skills.json`、8種）: `skill_<id>` フラグとして既存のイベント条件・重み付けに接続。能力値そのものは変えず、特定場面での選択肢解禁・重み補正に限定
- **制約・代償**（`game-data/burdens.json`、6種）: `burden_<id>` フラグ。`silver_purse`（初期資金+20だが強盗対象になりやすい）と `sealed_ring`（魔法・魔王系イベントを解禁するが危険を伴う）は `requiresBurden: true` とし、制約を1つ選ばない限り持ち込めないトレードオフを設けた
- **付与の適用・消費管理**（`game-engine/starting-grants.js`）: `applyStartingGrants` が転生開始時に一括適用し、`consumeItem` / `markItemUsed` が使用・消費・初回使用年齢を記録する
- **選択肢の解禁と封印**: 従来は `contextWeights` によるソフトな重み付けしかなく、アイテムを持たない転生者でもアイテム専用の選択肢を選べてしまうバグがあった（`illness_common` の看病選択肢が非所持者にも約79%の頻度で発生）。`decision-engine.js` に `requiredFlags` / `excludedFlags` によるハードな選択肢封印（`isChoiceUnlocked` / `unlockedChoices`）を追加し、該当選択肢に `requiredFlags: ["item_medical_kit"]` を設定して修正した
- **間接効果アイテムの使用記録**: 万能ナイフ・現代の教本・方位磁針・家族写真の4種は専用イベントを持たず `contextWeights` だけで人生へ影響するため、当初は実際に選択へ影響していても死亡時要約・観測画面で「未使用」のまま表示される不整合があった。`starting-grants.js` の `recordContextualItemUse` が、選ばれた選択肢の `contextWeights` に自身の持込アイテムのキーがあり、かつそのフラグが実際にコンテキストへ立っていた場合（=単に候補になっただけでなく、その選択肢が実際に選ばれた場合）にのみ初回使用を記録するよう `time-processor.js` の共通ループへ接続し、修正した
- **UI**（`index.html` / `web-ui/main.js`）: 転生開始画面を 候補概要 → 特殊要素・性格・能力 → 転生準備（アイテム/スキル/制約） → 選択内容の要約 → 転生開始ボタン の順に整理。各選択肢はカード形式で名称・一文説明・主な恩恵・想定される危険を表示し、内部IDや数値は見せない。制約を選ぶと `requiresBurden: true` のアイテムが解禁される
- **使用結果の意味モデル（`character.itemOutcome`）**: 当初は「初回使用年齢が記録された」ことだけを「役立てた」と表示していたため、銀貨袋を盗まれた（喪失）場合や指輪の封印を解かなかった（拒絶）場合まで「役立てた」と逆の意味で要約されるバグがあった。`character.itemOutcome = { status, age }`（`unused`/`used`/`consumed`/`lost`/`rejected`）を新設し、`setItemOutcome` を唯一の書き込み口として、`effects.itemOutcome: { itemId, status }` で肯定/否定の結果を明示できるようにした（`lose_purse` は `status: "lost"`、`leave_ring_sealed` は `status: "rejected"`）。`summary-generator.js` / `web-ui/main.js` の表示もこの状態を一次情報として参照するよう修正し、`consistency.js` に自己テスト（`runItemOutcomeSelfTests`）と人生ごとの検証（`findItemOutcomeViolation`）を追加した
- **人生ログ・死亡時要約**（`summary-generator.js`）: 与えたアイテム・スキル・制約、初めて活用した年齢、消費・喪失・拒絶の結果、最後まで使わなかった場合の明記までを死亡時要約に含める
- **セーブデータ**: `schemaVersion` を3→5へ拡張し、`web-ui/main.js` の `migrateState`/`migrateCharacter` が1〜4の既存セーブを5へ移行する（`startingItem`/`startingSkill`/`burden`/`itemState`/`itemFirstUsedAge`/`itemOutcome` を補完。schemaVersion4以前は喪失/拒絶を区別できていなかったため、既に接触済みのデータは安全側の `used` として引き継ぐ）
- **バランス検証**: `runBatchSimulation` の `grantStats` でアイテム・スキル・制約それぞれの選択数・使用率・平均寿命を集計。`runGrantComparisonTrial`（同一候補テンプレートを複製し、付与内容だけを変える比較試験）で、単一の付与要素だけがアーク到達率を80%超に押し上げていないかを確認する

## 実プレイで見つかった意味的矛盾の修正（issue #10, #11）

### 不老不死達成後も通常死亡する矛盾（issue #10）

`goal_check_immortality` の `immortality_reached` は、生涯目標「不老不死になる」を `completed` にするだけで、死亡判定へは一切反映していなかった。そのため達成後も老衰・病気などで通常どおり死亡し、「不老不死を達成したのに61歳で病死した」という意味的矛盾が起きていた。

複雑な不死継続モードを新設するのではなく、issueで最も安全とされた「不老エンディング」方式を採用した。`immortality_reached` の効果に `effects.endLife: { cause: "immortality_ascension" }` を追加し、`time-processor.js` の `simulateYear` がこれを検出すると、その場で（通常の老衰・病気の死亡ロールを一切行わずに）観測を終了する。死因ラベルにも `immortality_ascension`（「不老不死への到達（観測終了）」）を新設し、死亡時要約・観測画面・過去人生一覧のいずれでも通常死亡と区別できるようにした。`consistency.js` に自己テスト・静的チェック（`immortalGoalWithoutEndLife`：将来 goalResolution だけ追加されて endLife の対応を忘れる再発を検出）・人生ごとの検証（`findImmortalityViolation`）を追加し、800人生シミュレーションで「達成後の通常死亡」「未達成での不老不死終了」がいずれも0件であることを確認した。

### 「暦」が数値的に巻き戻る現象（issue #11）

`time-processor.js` の `driftWorld()` が `Object.keys(initialWorld)` を無条件に反復しており、`world.yearEra`（暦、共有世界の通算年）まで「初期値0へ回帰すべき世界統計」として扱ってしまっていた。これにより暦が毎年わずかに0へ引き戻され（巻き戻り）、かつ浮動小数点のノイズが蓄積して小数化する（例: `暦45.2125772467283年`）という、issue #5で一度修正した「非数値統計キーの混入」と同じパターンのバグが別の関数に残っていた。

ドリフト対象を `stability`/`warThreat`/`demonThreat`/`religiousInfluence`/`techLevel`/`economy` の6項目に明示的に限定し（`WORLD_DRIFT_FIELDS`）、暦は一切ドリフトの対象外とした。`schemaVersion` を5→6へ拡張し、既存セーブに残った巻き戻り済み・小数化した暦を四捨五入で修復する移行処理を追加。UI表示（`genLine`/`wYear`）にも念のため丸め処理を入れた。`consistency.js` の自己テスト（`driftWorld` を100回連続で呼んでも暦が完全に不変であることを直接検証）と、人生ごとの検証（1年ごとに暦が厳密に+1整数で進んでいるかを毎年確認する `yearRollbackCount`）を追加し、1,500人生（数万年分の年次処理）で巻き戻り0件を確認した。実機ブラウザでも、25回の連続転生・複数の死亡/次の転生遷移をまたいで暦が一度も減少しないことを確認済み。

### 目標ごとの決着率について

「王になる」「魔王を倒す」「英雄として名を残す」「魔王になる」は、他の目標に比べて決着（到達）率が低めです。これは、これらが `hero_mark` / `demon_mark` のような稀にしか付与されない特殊要素や、政治アーク・魔王アークのような多段階の連鎖を経由しないと到達できない、意図的に「狭き門」の目標であるためです（issue #3 が目指す「伝説級は5〜10%」という設計と整合しています）。個々の目標の到達率を均一にそろえることは非目標とし、`game-data/events.json` にエントリーポイントを増やすかどうかは今後の判断とします。「魔法を解明する」だけは元々どの経路からも到達できず平均進捗0件という明確な穴だったため、専用の決着イベント（`goal_check_unravel_magic`）を追加して解消しました。

## 転生記録図鑑と発見・解禁システム（issue #9）

PR #8までで転生準備フェーズが成立した一方、各人生がほぼ独立しており、複数回遊ぶことで観測対象・選択肢・記録が蓄積する感覚が弱かった。過去の転生結果を「転生記録図鑑」として蓄積し、人生で実際に遭遇した内容に応じて転生準備項目が段階的に解禁される仕組みを追加した。

- **発見カウンタ（`game-engine/discovery.js`）**: `state.discoveries`（特殊要素・生涯目標・最終職業・死因・発生イベントID・実績タグの6カテゴリ）はセーブデータ全体で1つだけ持つ、生涯をまたいで蓄積し続ける状態。死亡時に `recordLifeDiscoveries` で加算するだけで、減らす経路が存在しない。過去人生一覧の20件トリムとは独立しており、記録を減らしても発見状態は失われない
- **転生準備項目の段階解禁**: `game-data/items.json` / `skills.json` / `burdens.json` の一部（医療箱・方位磁針・家族写真・封印された指輪、魔力感知・危機察知、呪われた運命）に宣言的な `unlockCondition`（`{category, id, minCount}` を `any`/`all` で組み合わせ）と `unlockHint`（表示用の解禁条件文）を追加。`isItemUnlocked` / `isSkillUnlocked` / `isBurdenUnlocked` が条件を評価し、`applyStartingGrants` に `discoveries` を渡すとエンジン側でも未解禁の付与を拒否する（保存データの改変やUIの見落としで未解禁項目が付与される経路を塞ぐ）
- **死亡時スナップショット（`buildLifeRecord`）**: 名前・出身・最終職業・特殊要素・生涯目標と結末・人生ランク・実績タグ・転生準備の持込と使用結果・世界への影響・人間関係・歴史的重要イベントをまとめた1件分の図鑑エントリを生成し、`state.pastLives` へ格納する
- **選択肢封印フォールバックの撤廃（敵対的検証で指摘）**: `decision-engine.js` の `unlockedChoices` が「全選択肢が封印されたら封印を無視して全選択肢へ戻す」フォールバックを持っていたが、解禁条件付き選択肢しか持たないイベントを今後追加すると非対象者でも封印済み選択肢を選べてしまう再発経路になるため撤廃した。`pickChoice` は該当年に何も起きなかったものとして `null` を返すようにし、`consistency.js` の `findEventsWithoutUnconditionalChoice` で全イベントが「常に選べる選択肢を最低1つ持つ」ことを静的に強制する
- **UI（`index.html` / `web-ui/main.js`）**: タブに「図鑑」を追加。過去人生一覧→人生詳細（戻るボタン付き）と、発見率・発見済み/未発見（`？？？`表示）の一覧を切り替えて表示。転生準備画面では未解禁の項目に解禁条件のヒントを表示し、320px幅でも一覧・詳細・戻る操作が問題なく行えることを確認
- **セーブ・互換性**: `schemaVersion` を6→7へ拡張し、`discoveries` を持たない既存セーブは空の発見状態から再スタートする（過去人生データ自体は破棄しない）

### 動作確認

- `node scripts/simulate-cli.js 1000` で自己テスト26/26 PASS、静的チェック（`常時解禁の選択肢を持たないイベント`含む）・人生ごとの検証すべて0件を確認

## 死亡ログ後も人生が継続する状態不整合の修正（issue #14）

issue #10（不老不死到達）の対応以降、人生アークの結末選択肢の一部が「生涯を終えた」という結果文を出しながら、実際には `character.alive` を変更する効果（`endLife`）を持たず、機械的には人生が継続してしまう状態不整合が新たに見つかった。issueは「表示文言を解析して状態を変更する」実装（死亡ログの文字列に応じて `alive` を書き換える等）を明確に禁止しているため、今回の修正は一切そのアプローチを取らず、以下の方針のみで対応した。

- **状態遷移漏れ（本当に死ぬはずなのに `endLife` が無かったもの）**: `arc_religious_climax` の `executed_as_heretic`（異端として断罪される）と `arc_rebel_climax` の `captured_executed`（捕らえられ処刑される）の2choiceに `effects.endLife: { cause: "crime" }` を追加した。いずれも結果文が明確に処刑・死を述べているにもかかわらず、`health` を減らすだけで実際には生存し続ける経路になっていた
- **文言誤り（実際には死んでいないのに死を思わせる言い回しだったもの）**: `arc_artisan_climax` の `beloved_master`、`arc_peace_climax` の `live_in_guarded_peace` / `slip_into_obscurity` の3choiceは、状態としては人生が継続するにもかかわらずラベル・結果文が「生涯を終える」等の完了・終焉を思わせる言い回しになっていたため、地位確立・生活継続を表す文言に書き換えた（`endLife` は追加しない）
- **修正の副作用（自己発見・即修正）**: 実機ブラウザでの検証中に、`executed_as_heretic` の死亡要約が不老不死到達専用のはずの「老いや死の理から離れ…」という言い回しになっていることを発見した。これは `summary-generator.js` が `deathInfo.special`（＝通常の死亡ロールを経ずに `endLife` で終了したことのみを意味するフラグ）だけを条件にしていたため、処刑のような他の `endLife` 原因にも不老不死専用の文言が誤って適用されてしまっていたもの。`deathInfo.cause === 'immortality_ascension'` の場合のみ特別な言い回しを使うよう条件を絞り、処刑などその他の `endLife` は通常の死亡と同じ文言（「〜によりその生涯を閉じた。」）に統一した

## 既存セーブ移行時の発見状態復元（issue #15）

issue #9（PR #13）でschemaVersion 6→7へ拡張した際、`discoveries`（転生記録図鑑の発見状況）を持たない既存セーブは空の状態から積み上げ直す仕様にしていた。これは保存データそのものを壊すわけではないが、複数人生を既に観測済みのプレイヤーから見ると、過去人生一覧（`pastLives`）は残っているのに発見率・段階解禁だけが「未発見」へ巻き戻って見える、移行体験として不自然な状態だった。

`discovery.js` に `restoreDiscoveriesFromPastLives(discoveries, pastLives, occupations, deathCauseLabels)` を追加し、`main.js` の `migrateState` がschemaVersion 6以前から7へ初めて移行する瞬間（＝`discoveries`が存在しなかった移行）にのみ、既存の`pastLives`（最大20件）から復元可能な範囲を発見カウンタへ積算するようにした。

- **過去人生の記録形式によって復元できる範囲が異なる**: schemaVersion 7で導入された詳細レコード（`buildLifeRecord`が生成、`elements`フィールドを持つ）は職業・死因・特殊要素・生涯目標・実績タグをIDで直接復元できる。schemaVersion 6以前の簡易レコード（`elements`を持たない、職業・死因が表示ラベル文字列のみ）は、ラベル→IDの逆引きテーブルを組み立てて職業・死因のみ復元し、逆引きできなかった場合は捏造せず無視する
- **イベント発生履歴(`events`カテゴリ)は復元しない**: どちらの記録形式にもイベントIDそのものは保存されておらず（歴史的ログは表示用テキストのみ）、推測で埋めることはできないため、`events` カテゴリはこの復元では一切変更しない（issueが明示的に許容している欠落）
- **二重加算しない**: 復元処理は「discoveriesが存在しなかった移行」というスキーマバージョン境界の一度限りの分岐内でのみ呼ばれる。移行後はschemaVersionが7になり`discoveries`も存在するため、同じセーブを何度読み込んでも再実行されない
- **既存のschemaVersion 7セーブ（既に`discoveries`を持つもの）は一切変更しない**

## 保存・移行・初期化の安全性の可視化（issue #23）

`schemaVersion` の移行処理自体は既に実装されていたが、プレイヤーからは「保存されたか」「移行されたか」「初期化で何が消えるか」が画面から確認できなかった。長期プレイ前提のゲームとして、保存の信頼性をUIでも支えるようにした。

- **最終保存時刻・保存状態の常時表示**: ヘッダーに `#saveStatusLine` を追加し、`saveState()` の成否に応じて「最終保存: 14:32:10」または「保存失敗（この端末には保存されていません）」を常時表示。世界タブの「最終保存」（`wSaved`）も、未保存時は日時ではなく「未保存」と表示するよう修正（従来は保存前でも `Date.now()` を初期値にしていたため、実際には保存されていない時刻がそれらしく表示されていた）
- **旧schemaからの移行通知**: `loadState()` で読み込んだ生データの `schemaVersion` が現在の `SCHEMA_VERSION` 未満だった場合、「セーブデータを最新の形式に更新しました。」を通知する
- **保存データ破損時のフォールバック通知**: 不正なJSON文字列、オブジェクトとして壊れた構造（`schemaVersion` が範囲外、能力値欠落など）のいずれも、新規開始にフォールバックした理由を通知する（それぞれ別の文言で区別）。保存データが単に存在しない（初回プレイ）場合は、これらの通知を出さない
- **複数通知の並立**: 従来の `showBanner()`（既存の通知を消して1件だけ表示）に加えて `addBanner()`（消さずに積み増す）を新設し、起動直後に「移行通知」と「オフライン経過通知」が同時に発生しても、片方が片方を消してしまわないようにした
- **初期化ボタンの誤操作防止**: ブラウザ標準の `confirm()` を、削除される内容を具体的に列挙する確認カード（オーバーレイモーダル）に置き換えた。現在観測中の転生者、これまでの転生数、保存されている過去人生の記録件数、転生記録図鑑の発見状況、世界の状態が削除される旨を明示し、「初期化する」（危険色の専用ボタン）を明示的に押さない限り実行されない。「キャンセル」ボタンとEscapeキーの両方で閉じられる
- **保存失敗時のバナー抑制**: 自動再生中は毎年 `advanceOneYear` → `saveState()` が呼ばれるため、保存失敗の警告バナーを毎年積み増すと自動再生中に大量発生してしまう。保存成否が「成功→失敗」に変化した瞬間だけバナーを出し、失敗が続いている間は追加しない設計にした（ヘッダーの常時表示は失敗が続く限り「保存失敗」のまま表示され続けるため、状態自体は常に確認できる）
- **`localStorage` が使えない環境でも画面が壊れない**: 読み込み・保存・初期化時の削除は元々try/catchで包まれていたが、`window.localStorage` プロパティへのアクセス自体が例外を投げる環境（プライベートブラウズの一部実装など）でも、読み込み・保存・ゲームプレイが（保存されないことを除き）そのまま継続できることをPlaywrightで直接検証した

### レビューで指摘された初期化の削除失敗の扱い・起動時アクセス失敗の状態同期・モーダルのフォーカストラップ

初回実装には次の3点の不整合があるという指摘を受け、修正した。

- **初期化操作が `localStorage.removeItem()` の失敗を握りつぶし、常に成功扱いにしていた問題**: `removeItem()` の成否に関わらず `lastSaveOk = true` を強制していたため、削除に失敗しても画面上は初期化済みに見え、再読み込みすると古いデータが復活する状態だった。`removeItem()` の呼び出し自体をやめ、`state = freshState()` を `saveState()` で `STORAGE_KEY` へ上書き保存する設計へ変更。書き込みが失敗した場合は `saveState()` 既存の失敗処理（`lastSaveOk = false`、ヘッダーの「保存失敗」表示）がそのまま働き、加えて「画面上は初期化されましたが、保存領域への書き込みに失敗しました。再読み込みすると元のデータが復元される可能性があります。」という初期化固有の警告を追加した
- **`loadState()` で `localStorage` へのアクセス自体が失敗した場合に `lastSaveOk` が同期されない問題**: `getItem()` 自体が例外を投げるケース（アクセス不能）と `JSON.parse()` だけが例外を投げるケース（アクセスはできるが保存データが壊れている）を区別し、前者の場合のみ `lastSaveOk = false` を直ちに設定するよう修正。ヘッダーが「まだ保存されていません」という誤った表示のままにならないようにした
- **初期化確認モーダル（`role="alertdialog"`）にフォーカストラップが無く、Tab/Shift+Tabでオーバーレイの背後（見えない画面）へフォーカスが抜けてしまう問題**: モーダル内の可視な操作対象は「キャンセル」「初期化する」の2ボタンのみであることを踏まえ、`keydown` ハンドラでTabの両端を折り返すフォーカストラップを実装した

## 人生ログのタイムライン化（issue #24）

人生ログは年齢・暦順に並んではいたものの、生涯目標の形成・進捗・決着、職業や人間関係の変化、世界への影響といった「転機」が、他の細かい出来事の中に埋もれて見分けづらかった。また死亡後は要約テキストしか見られず、その評価の根拠になったログを死亡画面から読み返す手段がなかった。

- **転機を機械的に判定して構造化する**: `effect-processor.js` の `applyEffects`/`applyGoalFormation`/`applyGoalResolution`/`applyRelationEffect` を、それぞれ「実際に何が変化したか」（`changes`）を返すように変更した。テキストのパターンマッチではなく、エンジンが確定的に把握している変化だけを転機として扱うため、判定漏れ・誤検出が起きない
- **`time-processor.js` がログエントリへ `turningPoints` を付与**: `changes` から `{ type: 'goal'|'occupation'|'relation'|'world'|'death'|'birth', ... }` の配列を組み立て、各ログエントリに追加する。既存セーブのログエントリにはこのフィールドが無いが、UI側は常に `l.turningPoints || []` として読むため保存形式の互換性は壊れない
- **関係の微細な好感度変化はノイズとして除外**: `applyRelationEffect` が転機として報告するのは追加・離別・格上げのみで、affinityの微増減（毎年のように起こる背景の揺らぎ）は含めない
- **世界影響は死亡時要約・観測画面と同じ閾値を共有**: `summary-generator.js` に `WORLD_IMPACT_THRESHOLD`（=12）を切り出し、1イベント単体の増減がこの閾値以上の場合だけをログ上の転機として扱う。死亡時要約（`worldImpactPhrase`）・観測画面（`worldImpactChips`）・ログのタイムラインが同じ基準を共有するため、「ログでは転機扱いなのに要約には出てこない」という食い違いが起きない
- **人生記録タブでの見た目**: 転機を含むログエントリには通常より太い左ボーダー（`.logentry.turning`）と、種類ごとのバッジ（例:「目標達成「世界を救う」」「転職: 無職→冒険者」「出会い: アリア（仲間）」「世界: 王国の安定↑」）を表示する。既存の重要度フィルタ（すべて/転機以上/歴史的）はそのまま維持し、フィルタで絞った上でも転機がひと目で分かるようにした
- **死亡画面から生涯ログを振り返れるように**: 死亡画面の要約テキストの下に、既定で折りたたまれた `<details>`「この生涯の記録を振り返る」を追加し、開くと出生から死亡までのログ全体を時系列（古い順）で表示する。要約が「なぜそう評価されたか」を、同じ画面から遡って確認できるようにした
- **転生記録図鑑の過去人生詳細にも転機を継承**: `buildLifeRecord`（`game-engine/discovery.js`）が生成する `historicEvents` に `turningPoints` を含めるとともに、`turningEvents`（後述）を過去人生詳細で時系列マージして表示することで、死亡から時間が経った後の過去人生一覧・詳細でも同じバッジ表示を再利用できるようにした

### レビューで指摘された「過去人生化した時点でmajor/minorの転機が失われる」不具合の修正（issue #24）

## 自動再生・速度変更・オフライン経過の体感の調整（issue #19）

issue #19が挙げる5要件のうち、「自動再生中/停止中が明確に分かる」「オフライン経過で何年進んだか・主要出来事を通知する」「死亡/特殊エンディングでオフライン進行を止める」の3つは、既存実装（`timeState`/`playPauseBtn`のissue #17対応、`catchUpOffline`の元々の実装）で既に満たされていた。残る2要件──「速度変更の意味を短く表示する」「オフライン進行上限の意味をUI上でも説明する」──に絞って対応した。

- **速度の意味を短く表示**: `SPEEDS`の各エントリに`desc`（例:「1年 ≈ 3秒」）を追加し、選択中の速度の下に表示する`#speedDesc`要素を新設。速度ボタンをクリックするたびに更新される
- **オフライン進行上限の説明**: 「時間進行」カードに、上限の存在と値を常時説明する文言を追加した（詳細は下記レビュー対応を参照。初回実装時の文言は不十分だった）
- **上限到達時の個別通知は実装していない（意図的な判断）**: `mortality.js`の`mortalityChance()`は`age >= 100`で必ず1を返し（`Math.random()`の結果によらず`Math.random() < 1`は常に真）、`character.age`は`simulateYear`で毎年必ず+1されるため、誕生（age=0）から数えて約100年以内に必ず死亡する。`MAX_OFFLINE_YEARS`（300）はこれより大きいため、「上限に達したため打ち切った」状況は現在のゲームの死亡則の下では到達不能であり、その分岐を実装するとテストで検証できない死んだコードになる。そのため、上限の意味はUI上の常時表示でのみ説明し、オフライン経過バナー側には到達しえない特別分岐を追加していない

### レビューで指摘された「速度がオフライン換算にも使われる」ことの説明不足の修正

初回実装の説明文言（「離れていても最大300年まで自動で経過」）は、`catchUpOffline()`が不在時間(ms)を「選択中の速度のyearMs」で年数に換算するという実際の挙動を説明できていなかった。例えば「速い」（1年≈0.9秒）を選んだまま約90秒離れるだけで約100年分が処理され、通常はその途中で死亡する。300年という到達しにくい数値だけを強調すると、むしろ安全側の上限であるかのように誤解させ、issueの核心である「勝手に進みすぎた」という事故感を防げていない、という指摘を受けた。

- 「時間進行」カードの説明文言を、選択中の速度ラベル・その速度がオフライン換算にも使われること・上限年数・死亡/特殊結末で停止することの4点を含む文へ変更し、速度ボタンをクリックするたびに更新されるようにした
- 上限年数（300）をindex.html側に固定文言として持たず、`MAX_OFFLINE_YEARS`定数からJS側で文言を組み立てるようにし、表示値とロジックの値が乖離しないようにした
